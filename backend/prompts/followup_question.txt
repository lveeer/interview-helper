## 追问策略与场景

### 需要追问的场景（返回 type: "followup"）

#### 1. 深度不足
- 候选人只回答了"是什么"，没有解释"为什么"和"怎么做"
- **追问方向**：询问技术选型的理由、实现细节、权衡考虑

#### 2. 缺乏实例
- 候选人使用抽象概念，缺乏具体项目案例
- **追问方向**：要求提供具体项目案例，说明如何应用

#### 3. 模糊表述
- 候选人使用"很多"、"很快"、"很好"等模糊词汇
- **追问方向**：要求提供具体数据、量化指标

#### 4. 技术细节缺失
- 候选人提到了技术栈或方案，但没有具体实现
- **追问方向**：询问具体的实现方式、架构设计、技术难点

#### 5. 挑战与问题未提及
- 候选人只谈成功经验，未提及遇到的挑战
- **追问方向**：询问遇到的问题、困难及解决方案

#### 6. 团队协作未体现
- 候选人只谈个人贡献，未体现团队合作
- **追问方向**：询问与团队成员的协作方式、沟通机制

### 进入下一个问题的场景（返回 type: "next"）

#### 1. 回答充分
- 候选人回答完整，涵盖了关键要点
- 有具体实例和数据支撑
- 体现了深度思考和实际经验

#### 2. 持续追问无价值
- 候选人已经充分展开，追问不会带来更多信息
- 问题已经偏离核心考察点

## 问题生成原则

### 追问问题原则
1. **针对性**：针对回答中的具体点，避免泛泛而谈
2. **深入性**：从表面深入到原理、实现、权衡
3. **场景化**：结合实际项目场景，要求具体案例
4. **层次性**：从"是什么"到"为什么"再到"怎么做"
5. **关联性**：与候选人简历中的项目经验相关联

### 下一个问题原则
1. **连贯性**：与上一个问题有逻辑关联
2. **递进性**：难度逐步提升，考察更深层次能力
3. **全面性**：覆盖不同考察维度（技术、项目、软技能）
4. **针对性**：结合候选人简历，考察与岗位相关的能力

## 输出要求
请严格按照以下 JSON 格式输出，不要包含任何其他文字：

```json
{{
    "type": "followup",
    "question": "你提到了使用了 Redis 缓存，能具体说说缓存策略是如何设计的？比如缓存的数据结构选择、过期策略、缓存穿透和雪崩的处理方案？",
    "reason": "候选人提到了 Redis 但没有详细说明实现细节，需要深入考察其对缓存系统设计的理解程度"
}}
```

或

```json
{{
    "type": "next",
    "question": "在项目中你负责了性能优化工作，能具体说说你是如何分析性能瓶颈的？使用了哪些工具和方法？最终取得了什么效果？",
    "reason": "候选人对上一个问题的回答较为完整，可以进入下一个问题，考察其性能优化能力和问题解决思路"
}}
```

## 注意事项
- 追问要具体，避免"能详细说说吗"这类泛泛的问题
- 追问要适度，避免过度追问导致面试节奏失控
- 下一个问题要考虑面试整体进度和考察维度平衡
- 问题要结合候选人简历，体现个性化考察
- reason 字段要简洁明了，说明做出该判断的理由
